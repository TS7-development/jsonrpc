storage example 

Demonstrates how to store and call {\ttfamily print} with different types, which is working fine with the boost\+::any type.

Advantage of that solution is, that there is no need to adjust that \mbox{\hyperlink{structts7_1_1jsonrpc_1_1playground_1_1any__collection_1_1_storage}{Storage}}, if a new type is added since boost\+::any can store any type. In exchange the \mbox{\hyperlink{structts7_1_1jsonrpc_1_1playground_1_1any__collection_1_1_storage}{Storage}} needs to be adjusted for every new member function that shall be suppored, which happens way less than new types.

{\footnotesize\ttfamily void ts7\+::jsonrpc\+::playground\+::any\+\_\+collection\+::variant\+\_\+storage (\begin{DoxyParamCaption}{ }\end{DoxyParamCaption})}



variant\+\_\+storage example 

This example demonstrates how to store elements by using boost\+::variant.

This has worked quite fine. But compared to boost\+:any example \mbox{\hyperlink{namespacets7_1_1jsonrpc_1_1playground_1_1any__collection_ad8ccdabd8696851b7ba2b21fbc64d17b}{storage}}, this has some problems. First we always need to update \mbox{\hyperlink{namespacets7_1_1jsonrpc_1_1playground_1_1any__collection_a8a31e82446264021c7fcd0dcfc9f1879}{variant\+\_\+type}} to support every new type (i.\+e. every template combination for a JSON RPC callback signature). Secondly, for every new method that shall be callable on those type, we also need to implement a new visitor. This is basically the same workload as with boost\+::any that requires a new callback lambda for every new supported member function.
