storage example 

Demonstrates how to store and call {\ttfamily print} with different types, which is working fine with the boost\+::any type.

Advantage of that solution is, that there is no need to adjust that \mbox{\hyperlink{structts7_1_1jsonrpc_1_1examples_1_1any__collection_1_1_storage}{Storage}}, if a new type is added since boost\+::any can store any type. In exchange the \mbox{\hyperlink{structts7_1_1jsonrpc_1_1examples_1_1any__collection_1_1_storage}{Storage}} needs to be adjusted for every new member function that shall be suppored, which happens way less than new types.

{\footnotesize\ttfamily void ts7\+::jsonrpc\+::examples\+::any\+\_\+collection\+::variant\+\_\+storage (\begin{DoxyParamCaption}{ }\end{DoxyParamCaption})}



variant\+\_\+storage example 

This example demonstrates how to store elements by using boost\+::variant.

This has worked quite fine. But compared to boost\+:any example \mbox{\hyperlink{namespacets7_1_1jsonrpc_1_1examples_1_1any__collection_a69a2fe8952fdd0d2de1790af65310c95}{storage}}, this has some problems. First we always need to update \mbox{\hyperlink{namespacets7_1_1jsonrpc_1_1examples_1_1any__collection_ad04effe47face3f4ae870582cb91ef8b}{variant\+\_\+type}} to support every new type (i.\+e. every template combination for a JSON RPC callback signature). Secondly, for every new method that shall be callable on those type, we also need to implement a new visitor. This is basically the same workload as with boost\+::any that requires a new callback lambda for every new supported member function.
